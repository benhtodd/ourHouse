In this lab, we are going to put a DMZ constraint to our blueprint for the networks. We are going to examine how these affect the deployment engine decisions for resource placement.

Adding DMZ Network Capability Constraint

Open the Tito - Placement - yourname blueprint. Add/Uncomment the following Constraint Tag for the app-public-net network:

       - tag: 'cas.network.cap:dmz'

Deploy the blueprint with default inputs.

Examine the deployment request details from the Requests tab for more info.

Why did the deployment engine decide to place the deployment in PROD?

DO NOT delete the deployment at the end of the lab.

HINT 1

Answer is on the Network Profiles tab.

Solution: Adding DMZ Network Capability Constraint

From the previous lab, we saw that after adding the PCI constraint the available regions for deployment were TEST and PROD.  TEST was the preferred choice by the deployment engine as it has higher priority for the project.

After adding the DMZ constraint we limited the deployment option to only PROD. If you examine the network profiles you will notice that only the prod profiles have the cas.network.cap:dmz capability tag.

DO NOT delete the deployment at the end of the lab.
