In this lab, you'll be logging into Cloud Assembly and exploring the setup which was configured for you. 

Exploring the Configuration

Please do not change any of the capability tags assigned to different objects throughout Cloud Assembly. By doing so you can break the labs for all students!

Let's begin by opening a browser window to https://console.cloud.vmware.com and logging in. You have already logged into the VMware Cloud Services Portal once before. 

Make sure to navigate to the Livefire Labs org.

  1. Start by opening the VMware Cloud Assembly under My Services.
  2. Explore the following configuration settings:
  3. Navigate to the Infrastructure tab and then click Cloud Accounts. View both AWS and Azure accounts.
    1. Make note of the Regions we have selected.
    2. Make note of the Capabilities tags.
  4. Navigate to the Cloud Zones tab. View at least one AWS and one Azure zone.
    1. Make note of the Capability Tags that are set on the Summary tab. You will notice that a zone has either AWS or Azure tag which identifies to which cloud account the zone is going to deploy it. It also has a dev, test or prod tag which tells us that Elastic Sky company considers this cloud zone (region) either test, dev or production one.  
    2. Make note of the computes added on the Compute tab. This shows us on which computes within this zone we can provision to.
  5. Navigate to the Flavor Mapping tab. View at least one of the Flavor mappings. We will be using mainly the Micro mapping for the purpose of the labs. This shows us what VM sizing we consider Micro in each region for each cloud account.
  6. Navigate to the Image Mapping tab. View at least one of the Image mappings. The Image Mapping profile shows which cloud OS image we are going to use for each region to provision resources. We will be using mainly the centos-74-image mapping. The centos-74-livefire-image and the ubuntu-16-livefire-image mappings are custom cloud images created by the Livefire team. Explore at least one more mapping that contains native cloud images.
  7. Navigate to the Network Profiles tab. View at least one AWS and Azure network profile.
    1. Make note of the Capability Tags that are set on the Summary tab.  The tags used here identify what Elastic Sky considers a dev, test or prod network profile.
    2. Make note of the networks under the Network tab. Make note of the Capability Tags applied to those networks. These tags help us identify what Elastic Sky considers a public or private network and where these are coming from: AWS or Azure.
  8. Navigate to the Storage Profiles tab. View at least one AWS and Azure profile.
    1. Click on at least one storage profile per cloud zone and examine the configuration.
    2. Make note of the Capability Tags applied to those profiles. The tags here identify what Elastic Sky considers PIC, Bronze, Silver and Gold storage profiles.
  9. Navigate to the Projects tab.
    1. Open dev-tito.
    2. Navigate to the Provisioning tab and make note of the configuration. More precisely, make note of the Cloud Zones configuration and priority that is set.
  10. Feel free to examine the rest of the interface at your leisure.
Solution: Exploring the Configuration

Navigate to the Infrastructure tab and then click Cloud Accounts.

View both AWS and Azure accounts.

Make note of the Regions we have selected. Make also note of the Capability tags applied. We have a capability tag that describes the type (AWS/Azure/...) of the account and the customer (owner) to which it belongs.  If needed we can easily use these as constraints to target deployments to a particular type of cloud account or cloud account that belongs to a particular customer/owner.

Navigate to the Cloud Zones tab. View at least one AWS and one Azure zone.

Make note of the Capability Tags that are set on the Summary tab.

 You will notice that a zone has a dev, test or prod tag which tells us that Elastic Sky company considers this cloud zone (region) either test, development or production one.  

Make note of the computes added on the Compute tab. This shows us on which computes within this zone we can provision to.

View at least one of the Flavor mapping. We will be using mainly the Micro mapping for the purpose of the labs. This shows us what VM sizing we consider Micro in each region for each cloud account.

We will be using mainly the  centos-74-livefire-image mapping. It is a custom cloud image created by the Livefire team. Explore at least one more mapping that contains native cloud images.

Navigate to the Network Profiles tab. View at least one AWS and Azure network profiles.

Make note of the Capability Tags that are set on the Summary tab.

The tags used here identify what Elastic Sky considers a dev, test or prod network profile.

These tags applied here help us identify what Elastic Sky considers a public or private network and where are these coming from: AWS or Azure.
