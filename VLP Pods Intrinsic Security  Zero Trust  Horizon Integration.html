1. vCloud Director vAppTemplate / vPod

The vAppTemplate you wish to use for delivery should be in the Livefire-Delivery-Catalog as per:

How To: Copy vAppTemplate to Delivery Org

1.1. Update the vAppTemplate (if necessary)

NOTE: This is ONLY required if you have just updated your vAppTemplate and have a new version in the Livefire-Delivery-Catalog

Log in to the VMware Learning Platform - livefire tenant

  1. Make sure that Advanced Options is enabled
  2. Expand Cloud Management
  3. Select Clouds
  4. Click the triple-dot menu
  5. Select Full Resync vApp Template
  6. Verify that your updated vApp Template is now visible by checking in vApp Templates

These steps will update VLP's knowledge of what vAppTemplates are currently available in the Livefire-Delivery-Catalog

Now we must update the Deployment Pool to use the updated vApp Template:

  1. Click Deployment Pools
  2. Find the appropriate Deployment Pool that needs to be updated, and click the triple-dots to access the menu (There should be one deployment pool per VLP class. As of this manual, these are: "Digital-Workspace-Zero-Trust" and "LSS-IS-Intrinsic-Security"
  3. Click Edit vApp Templates

DO NOT DO THE NEXT STEP DURING AN ACTIVE CLASS OF THE SELECTED DEPLOYMENT POOL

  1. Delete (or optionally Disable) the currently specified Template
  2. Uncheck the VM Equality Check box
  3. Click the UPDATE DEPLOYMENT POOL button
  4. Click the ADD VAPP TEMPLATES button
  5. In the Add vApp Templates window, select the updated vApp Template and
  6. Click the ADD button
  7. Click the UPDATE DEPLOYMENT POOL button

The next scheduled class wil now use the updated vApp Template

1.2. Schedule Intrinsic Security Class

The steps below show how to schedule the Intrinsic Security Class lab resources.

Log in to the VMware Learning Platform - livefire tenant

  1. Click on Classes (Advanced Options not required)
  2. Locate the last instance of the Intrinsic Security Livefire "Class" and click the triple-dots to activate the menu
  3. Click Duplicate

On the Basic Details step of the Create Class dialog;

  1. Keep the class name of Intrinsic Security Livefire (Optionally update if the class title has changed. This title will appear in e-mail notifications)
  2. Update the Unique Class ID: [GEO]-ddMMMYY-islf (For example: amer-3may21-islf) The suffix is essentially the initials of "Intrinsic Security Livefire"
  3. Add the assigned (and supporting) Instructors as needed. Include Burke & Erika as well please
  4. (OPTIONAL) Include description of the class. This REALLY isn't required and is typically left blank
  5. Click NEXT

No changes should be required on the Lab Details step since the Intrinsic Security Livefire lab should already be selected. If, for any reason, it is not then be sure it is selected!

  1. Click NEXT

On the Schedule step;

  1. Select the appropriate Timezone for the scheduled delivery
  2. Set the start Date and Time
  3. Set the End Date and Time to 1 DAY AFTER the actual scheduled delivery time (Ideally 8PM for the scheduled timezone. The screenshot shown above was actually for a US Central Timezone delivery.
  4. Click NEXT

WARNING: When the End Date/Time is reached, VLP will automatically SHUTDOWN and DELETE all the Entitlements. This results in all the vCloud Director vApps being that are associated with the entitlements get deleted AND the UEM Cleanup Callback is executed.

On the Students step;

  1. Set the What is the max limit for the class? value to 2 more than the confirmed number of attendees (This allows for a couple spare vApps)
  2. If testing, use the +ADD STUDENT to add your test account(s)
  3. Erika is responsible for Uploading the CSV for actual deliveries (she can always come in later and EDIT the class to upload the CSV, this is how the delivery in the screenshot was done)
  4. Click NEXT

On the Resource Mgmt step;

  1. Set the Warm-up labs to 23 Hours (cannot be more than 24 due to platform limitation)
  2. Set Keep additional to 2
  3. Disable Resource saving options
  4. Click NEXT

On the Publish step;

Click CREATE CLASS

Carefully review the Start Date and End Date of the class that you scheduled. The End Date can be a bit unpredictable! I had multiple issues with it not showing the actual selected date. If it does not show accurately, give it several minutes and then Refresh. If still an issue, update the class again to adjust the end date.

We DON'T want a class to accidentally END and clean itself up a day early! ! !

1.3. Provision Entitlements Earlier than 23hrs prior to class start (OPTIONAL)

NOTE: This only needs to be done if you wish to pre-deploy the class more than 23 hrs before class start. If you have a start day of Tuesday (and Monday is not a holiday), it may be a good idea to do this process on Saturday or Sunday prior to class in order to have the deployment CPU hit have minimal effect on the vCD Cluster.

The VLP system will automatically deploy all vApps 23 hours prior to the schedule start time (Assuming the directions in this guide were followed.) In some cases (such as the NOTE just above this text), it may be desirable to pre-deploy the pods even earlier. The process is actually pretty simple:

Log in to the VMware Learning Platform - livefire tenant

  1. Click on Classes
  2. Locate the class and click on the triple-dots to activate the menu
  3. Click Provision Entitlements

The screenshot above simply shows the view from the Delivery org in vCloud Director after a class has been provisioned via VLP

What Happens during Provisioning of the Entitlements (whether manual as shown here or scheduled):

  1. VLP will check the Specified Lab, Deployment Pool, and then deploy and power on a copy of the specified vAppTemplate for each of the confirmed attendees.
  2. Once the vApp has been provisioned, it is then immediately entitled which results in the vApp name in vCloud Director showing as VLP-LSS-IS-CLASS-NAME-NEE-ENTITLEMENT-ID-Attendee-Email
  3. As each of those Entitlements are set, VLP will then execute the UEM Callback automation

1.4. Get External Pod IPs

We use vRealize Automation and a custom vRealize Orchestrator Workflow to perform this step. This environment is running in our lab, but can be accessed from your local system if desired.

Log in to the Livefire Automation Portal - Service Broker ( https://vra.lab.livefire.dev/catalog )

If you are not already logged in, you will be redirected to woa.lab.livefire.dev - please select "LAB.LIVEFIRE.DEV' as your domain before logging in.

Locate and Request the Get <CLASS> IP... catalog item for the CLASS you have deployed

This catalog item will process all currently running vApps in the Livefire Delivery org that begin with VLP-LSS-???.

If the ??? = DW, then dwuser001 -> dwuser025 (or however many vApps are found) will be assigned to the users who are entitled to the vApps.

If the ??? is IS then isuser001 -> isuser025 (or however many vApps are found) will be assigned to the entitled users.

This information is gathered and formatted into an e-mail that is sent to the Instructor E-mail addresses that were entered by the person submitting the form

 

BEFORE YOU CONTINUE:

  • Be sure you have updated the isuser### password (This should have been done after the last class completed to prevent unauthorized access to the lab environment)
  • Be sure that the VLP Deployment process has completed - can be verified by logging in to the Livefire Delivery Org in vCloud Director

For class type, enter either: VLP-LSS-IS or VLP-LSS-DW <- No spaces before or after, must be EXACT

Add each of the Instructor E-mail addresses that should receive the list of Pod IP Addresses for all of the VLP Deployed vApps

Within a few minutes of submitting the request, the specified recipients should receive an e-mail similar to what is shown above.

Regarding the Horizon Login -- this is NOT bound in any way to the vApps/vPod IP/or the attendees at this stage. You can copy/paste the Login/Password/IP info to the Class Roster. The piece that IS BOUND is the vPod IP -> vApp and those vApps are tied to the attendee that it is entitled to.

DO NOT REASSIGN an existing vApp to another user as the result of a drop/cancel/no-show. Instead, END the entitlement for the drop/cancel/no-show and Add the replacement attendee in VLP in order for that person to have their own entitlement.

5. Reference Info for Adding New Deployment Pool

The primary purpose of Livefire using VLP for our deliveries is the Callbacks for the UEM integration. In order for the callbacks to work, there are couple requirements.

  1. We worked with the team to validate that our new "livefire" VLP tenant can make the necessary callback as long as the entitled vApps have the following Naming syntax: The Lab SKU Must Start with "LSS-DW", "LSS-IS", or "LSS-AW" so that the Entitlements Callbacks will function

    For Example, the AW Zero-Trust vApps, when entitled, are named VLP-LSS-DW-Zero-Trust-NEE-197562058023-user@example.com.
  2. The LF-airwatch callback item must be added to the VLP "Lab" under "Entitlement Callbacks"

In order to add a new delivery in VLP, start by making sure the vAppTemplate has been added to the Delivery Org and Sync'd to VLP.

Next, you must:

  1. Create a Deployment Pool
  2. Create a Lab

Then you are ready to create and schedule a class (Not going to document this as it is simply a create process nearly identical to the steps above for "Schedule Intrinsic Security Class")

5.1. Create a Deployment Pool

In order to make sure that vApps are properly named upon being entitled:

  1. Make sure that Advanced Options is enabled
  2. Expand Cloud Management
  3. Select Deployment Pools
  4. Click the + CREATE DEPLOYMENT POOL button
  1. Give the deployment pool a name and description
  2. Select the vApp Template that will be used for this delivery
  3. Click CREATE DEPLOYMENT POOL
  1. No changes required on the Edit vApp Templates page, click NEXT
  1. On the Edit VMs page, locate the Main Console/ControlCenter VM and place a check in the Initial console checkbox
  2. Click FINISH

5.1.1. Update the vApp Naming Template

In order to make sure that vApps are properly named upon being entitled:

  1. Make sure that Advanced Options is enabled
  2. Expand Cloud Management
  3. Select Deployment Pools
  4. Select the Deployment Pool that you wish to update
  5. Click the Triple-dots menu at the top right
  6. Select vAppNaming Template
  1. In the Naming Template box, use this EXACT String (this should be the default. If it is not, then update it):

VLP-[labSku]-[entitlementId]-[username]

  1. Click the VAPP NAMING TEMPLATE (or CANCEL if no changes required) button to save and close the window

5.2. Create a Lab

  1. Make sure the Advanced Options is enabled
  2. Start the SKU with "LSS-DW", "LSS-IS", or "LSS-AW" then add team abbreviation and class-type abbreviation or name. In this case, it is an "Anywhere Workspace Horizon Integration" lab so I specified: LSS-AW-HI -- this could also be changed to LSS-AW-Horizon-Integration
  3. Give the lab a Readable Name, such as: Anywhere Workspace Horizon Integration
  4. Select the Deployment Pool that will be used for this lab
  5. Set the Valid Period and Time Allotted to 12 Days 1 hour - this allows for the lab to be created the Thurs/Fri before a scheduled delivery
  6. Click CREATE LAB

REMINDER: SKU MUST match as follows in order for the callback to function!

The configured SKU wildcards for your team are:

· LSS-DW

· LSS-IS

· LSS-AW

If the Lab SKU doesn't match one of those, the callbacks won't work unless I add the SKU to the engine

Don't make any changes to the first half of the Update Settings page:

  1. Scroll Down

The only change required on this entire page is:

  1. Add the LF-airwatch, url=url callback to Entitlement Callbacks
  2. Click NEXT
  1. No changes are required on the Update Content page, click NEXT
  1. Place a check in the Use from Deployment Pool box for Initial Console
  2. Scroll down...
  1. Click the Use Deployment Pool value (False) for the Use individual VM power on settings
  2. Click FINISH
