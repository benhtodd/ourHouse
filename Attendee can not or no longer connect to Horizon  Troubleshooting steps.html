Attendees frequently have a variety of issues on their initial and/or subsequent connection attempts for the Horizon Desktops. Symptoms/Causes can include:

Individual Attendee: (may happen to one or more attendees/instructors)

  • Old Horizon Client
    • All attendees should be encouraged to install a more current version of the client - especially if currently running anything less than 8.0.0
    • The current Horizon Client version as of this article is 8.2.0
  • SSL Issues
  • General routing/connectivity (client/ISP side)
  • HTTP error 504 - attendee tried to reconnect too quickly

All Attendees / Instructors: (Happens to Many/All attendees/Instructors)

  • Unified Access Gateway (UAG) issue(s) - in the past, our UAGs disks filled up, preventing successful connection to Horizon Connection Servers and desktops
  • Connection Server(s) issue - we have not yet had an issue with the connection servers
  • Datacenter Connectivity - we have not yet had this come up as an issue

If Nobody (including Instructors) can access the Horizon Desktops (even after trying different VPN endpoints and/or Attempting RDP via our RDP Gateway), immediate escalation to lab team is required. If this becomes the case during delivery, the best thing to do is continue on with Lecture and/or provide iSIM content (or VMware Hands On Labs recommendations) for attendees to keep their time from being wasted while the issue is addressed with Lab team and/or Corporate HaaS team.

If several people CAN access the lab environment, and only some cannot, then the following steps should be taken:

Common Issues

1. The connection to the remote computer ended

This has happened in the past when the "Allow connection via SSL Proxy" option was not checked when it was supposed to be.. after checking the box and trying again, attendees can usually connect. This info is already covered in the Lab Manuals.

2. The connection to the remote computer has been closed due to a new connection request

This typically happens when a user connects to your desktop... For example, mduser010 is logged in and working... suddenly this message pops up and the user is disconnected. Meanwhile, another attendee (say mduser019, for example) logs in using mduser010 credentials accidentally.

3. Error: HTTP error 504 / Failed to connect to the Connection Server

This error occurs when a user disconnects (or is disconnected from) their Horizon Desktop and they attempt to immediately reconnect.

Solution: Patience; wait a minute or two and then reconnect

4. Assigned desktop source for this desktop is not currently available

This error most commonly occurs when the View Agent has stopped responding. This can be verified by checking the Horizon Management UI, Desktops view, then filtering on the desktop id.

  1. The Agent Version will display as Unknown
  2. The Status will display as Agent Unreachable

How to proceed:

Scroll down to: Instructor/Lab-Team check attendee desktop viewagent service and begin troubleshooting from that step.

5. Failed to resolve proxying route for request

This error applies only the the HTML Client and most commonly occurs on MacOS when using Safari or Firefox.

Solution: Use a Chromium based browser such as: Google Chrome, Chromium, or Microsoft Edge - this nearly always works

Encourage the use of the Installable Horizon Client since this is preferable and provides a better overall user experience (especially with copy/paste) than the HTML client

6. All available desktop sources for this desktop are currently busy

This error means that there aren't enough desktops in the Horizon Desktop Pool to meet the demand. This error was experienced by the NSX team when the pool was set to a maximum of 30 Desktops and a second class was launched concurrently. Since then, the max Desktops per Pool for Windows has been set to 75. The Ubuntu Pool maxes are set to 60.

7. Error putting route. Reason - Connection refused: localhost/127.0.0.1:8123

This was the connection error that was experienced when both of our UAGs had their disk fill up due to a log file consuming all disk space. This was determined to be an engineering error where the version of UAG we were using did not have logrotate configured for the 20.12 version of the appliance. After installing version 21.03, we found that a logrotate did exist. Full e-mail explanation was sent by Burke to the team on Monday, May 31st, 2021 after working all weekend on this (and other) lab issue.

8. HTTP error 429

This error seems to appear every once in a while randomly. When you click OK and try connecting to the desktop pool, the error appears again. In order to get logged back into the desktop:

  1. Disconnect from the Horizon Endpoint
  2. Re-Connect to desktop.livefire.dev
  3. Access Desktop Pool

General Troubleshooting Steps

1. Instructor/Lab-Team should attempt login as attendee

Instructor or Lab team should attempt to login as the attendee account using either the HTML client or the Installable client. This is the FASTEST way to determine if there is a potential issue that exists only on the Attendee side (IE: SSL Certs, mis-configured client, old client, internet (ISP) routing, firewall, etc...)

1.1. Instructor/Lab-Team can login: Client Side issue

If any instructor, lab-team, or other attendee member can login to the desktop a particular attendee cannot access, then the issue must be on the attendee's side:

  1. If possible, have attendee attempt to access https://desktop.livefire.dev from another device and/or network.
    For example, in a July 2021 class an attendee suddenly could not access the Horizon Environment via their computer on their home network. After switching their phone's wifi off and using phone to connect, they were successful. This isolated the connectivity issue to their ISP.
  2. Verify Client version >= 8.1.0
  3. All security settings good?
  4. Has the Horizon Client been uninstalled, rebooted, fresh install & configure?
  5. Is attendee on VPN? if so, try disconnecting?
  6. Can attendee try using VPN or Different VPN endpoint (IE: we have had German attendees suddenly unable to connect. After changing the VPN connection to a North America based VPN endpoint, they were able to reconnect)

1.2. Unable to login to the specified Desktop

If nobody can login to this particular desktop using the Horizon Client or the HTML Client, but CAN login to other desktops, then continue with troubleshooting steps below.

2. Instructor/Lab-Team check attendee desktop viewagent service

All instructor desktops can ssh to attendee desktops. Once connected, check the viewagent service to make sure it is still running:

sudo systemctl status viewagent --no-pager

This should show the "Active: (active (running)" like the screenshot above. If anything other than "active (running)" is output, try restarting the service:

sudo systemctl restart viewagent

3. Instructor/Lab-Team check Horizon Management UI

If nobody can login to a particular attendee desktop, the next step is to check the Horizon Management UI, Machines Page to see if the Agent Version is showing.

For instructions on logging in, restarting, resetting, and Recovering Horizon Desktops, see this article:
How To Login and Restart/Reset/Recover Attendee Desktop

If the "Agent Version" column shows anything other than a valid Version number, then the Desktop most likely needs to be Restarted (Guest Os Restart) or Reset (Guest OS Power-Off/On).

As a LAST RESORT:

If, even after a restart of the Desktop, the desktop still cannot be logged into, then the only remaining option is to "Recover" the desktop( Sets Desktop back to fresh base image ). This will result in the loss of any data files (configs, scripts, etc...) that have been created on that desktop. As an instructor, you could SCP those files to a temp folder on your Instructor desktop and then scp them back tot he attendee desktop once it has been recovered.
