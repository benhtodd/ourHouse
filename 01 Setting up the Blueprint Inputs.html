In this chapter we will Clone (or Import) a base Blueprint that we will use as a skeleton and build upon. We will also start configuring the Blueprint inputs to match our company needs.

Importing the Blueprint

  1. The Big Blueprint v8 (BB8) - Start.yaml should be available in the Design tab in Cloud Assembly. Alternatively you can also find it in Dropbox. If it is not there, ask your instructor(s) for the blueprint.
  2. Clone the existing blueprint or update the copy from Dropbox.
  3. In Cloud Assembly navigate to the Design tab to manage Blueprints and select Upload.
  4. Name your blueprint Big Blueprint v8 (BB8) - yourname and add it to the existing dev-tito  project.

Setting up the Blueprint Inputs

If you try to deploy the BP you will notice that , although the inputs have been defined, they are lacking a lot of properties. In this section will will modify the attributes to meet our company needs for the BP.

Let's start with the looking at the inputs and what our company requirements are.

flavor input requirements:

  • flavor input must be a dropdown field with a list of all available flavors.
  • Default choice listed should be the smallest flavor.

environment input requirements:

  • The Input must be a dropdown field with a list of all available cloud zone environments.
  • The dropdown should also present an option to select any of the available cloud zone environments.
  • Items in the dropdown list should use use friendly names for the cloud zone environments: e.g. Production, Test, ...
  • Default choice listed should be the Development environment.

We are giving you a hint with a preview of the environment input presentation. Most of the remaining inputs have similar presentation.

cloudZone input requirements:

  • The Input must be a dropdown field with a list of all available cloud zones types.
  • The dropdown should also present an option to select any of the available cloud zone types.
  • Items in the dropdown list should use use friendly names for the cloud zone types: e.g. AWS, Azure
  • Default choice listed should be AWS

network input requirements:

  • The Input must be a dropdown field with a list of all available network types.
  • The dropdown should also present an option to select any of the available network types.
  • Items in the dropdown list should use use friendly names for the cloud zone types: e.g. Public, Private
  • Default choice listed should be Public 

loadBalancer input should present the user with an option to deploy or not to deploy a load balancer. Later on when you work on the Load Balancer section in the BP you can revise these requirements if you cannot determine what should be configured now:

  • The Input must be a dropdown field giving a Yes/No list option to deploy or not to deploy a load balancer.
  • Default choice listed should be Yes 

storageTier input requirements:

  • The Input must be a dropdown field with a list of all available storage tiers.
  • The dropdown should also present an option to select any of the available storage tiers.
  • Items in the dropdown list should use use friendly names for the cloud zone types: e.g. Silver , Bronze, ...
  • Default choice listed should be Silver

resourceGroupType should enable the user to select between creating an on-demand Azure resource group (RG) for each deployment, use a predefined Azure RG for the deployment Region, or use an existing RG Group. Name must be provided in next field.  If existing RG group is used the name must be specified in the resourceGroupName input.

resourceGroupType input requirements:

  • The Input must be a dropdown field with a list that includes the following resource group types : predefined, ondemand, existing
  • Default choice listed should be ondemand

resourceGroupName input requirements:

  • The Input must be a string field where users will enter name for an Azure RG when the resourceGroupType is Existing
  • The input should have an empty default value
  • Valid value for the input should allow only: alphanumeric characters, periods, underscores, hyphens and parenthesis.

tagsOptional input requirements:

  • The Input must be an array of string objects.  
  • It must provide the user to optionally enter key value pairs that later on will be used in the BP and applied to all objects as tags.
  • The input should have no initial value.
  • When specifying the optional tags a description should be available for the Key and Value inputs. The description should provide the following information: Only hyphens (-), underscores (_), lowercase characters, and numbers are allowed.

tagsRequired input requirements:

  • The Input must be an array of string objects.  
  • As these are required tags, user must not be able to add, remove or modify these tags during deployment.
  • The input should contain 2 key value pairs: cas-resource-desc:class-delivery and cas-resource-source:vra-cloud

  • same as storageTier input

SOLUTION

The solution YAML representing all the resources part of the blueprint Big_Blueprint_v8_(BB8)_inputs.yaml can be found on the Dropbox share. Take a look at it and go back to the requirements to see how each end every requirement was met in the YAML code.

When comparing your code vs the solution code make sure to note:

  • How we've used oneOf property to provide the dropdown list with friendly names
  • How we've created an array of string objects for the Tag inputs.
  • How we've locked/made read only the tagsRequired input so that it cannot be changed by a user.
  • How we've used empty '' value for all inputs that we use as constraints for deployment. In this case no constraint is passed and the company requirement to be able to select "any of the" is fulfilled.
